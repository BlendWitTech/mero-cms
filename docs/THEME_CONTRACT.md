# Mero Theme Contract

The formal contract between a Mero theme and the rest of the system. If your theme satisfies everything in this document, the visual editor, capability gating, plugin store, and seed data flow all work without further configuration.

If you're new, start with [PARTNER_AGENCY_GUIDE.md](PARTNER_AGENCY_GUIDE.md) — this doc is the spec, not the tutorial.

---

## 1. The package

A theme is an npm workspace package living at `themes/<slug>/` with this minimum shape:

```
themes/your-theme/
├─ package.json          # name, version, depends on @mero/theme-base
├─ theme.json            # the manifest (this doc covers what's required)
├─ next.config.js        # standard Next.js config
├─ tsconfig.json
├─ src/
│  ├─ app/
│  │  ├─ layout.tsx      # wraps every page; mounts <EditorBridge />
│  │  └─ page.tsx        # home, plus per-route folders
│  ├─ components/        # your custom sections, if any
│  ├─ lib/
│  │  ├─ widget-registry.gen.tsx   # generated by `mero theme build-registry`
│  │  └─ widget-registry.tsx       # tiny re-export from .gen.tsx
│  └─ styles/
│     └─ globals.css     # CSS variables here, one source of brand truth
└─ public/               # static assets (logo, favicons, hero illustrations)
```

`@mero/theme-base` provides every default section component, the API client (`getSiteData`, `getPage`, `getMenu`, etc.), the capabilities provider, and the editor bridge. You import what you need; you don't have to recreate any of it.

## 2. `theme.json` — required keys

| Key | Type | Required | Purpose |
| --- | --- | --- | --- |
| `name` | string | yes | Human-readable theme name. |
| `slug` | string | yes | Matches the folder name. |
| `version` | semver string | yes | Major bump = breaking change to `pageSchema`. |
| `description` | string | no | Shown in the admin theme gallery. |
| `minPackageTier` | 1\|2\|3\|4 | no | Refuses activation on lower tiers. |
| `supportedPackages` | string[] | no | Allowlist of package ids, or `["any"]`. |
| `requiredCapabilities` | string[] | no | Refuses activation if any are missing. |
| `optionalCapabilities` | string[] | no | Theme renders enhanced features when present. |
| `widgetCatalog` | object[] | yes for visual editor | Drives the editor palette. |
| `widgetCategories` | object[] | no | Custom categories in the palette sidebar. |
| `moduleSchemas` | object | no | Field shapes for legacy section ids. |
| `pageSchema` | object[] | yes | Every page slug + its sections. |
| `seedData` | object | yes | Initial pages and content the activation seeds. |
| `brandingFields` | object[] | no | CSS variables the admin's branding tab can override. |

The validator (`npm run theme:validate -- <slug>`) checks all of these.

## 3. `widgetCatalog` — entry shape

```jsonc
{
    "type": "Hero",                         // PascalCase. Matches the registry import.
    "name": "Hero",                         // Shown in the palette.
    "description": "Big headline + dual CTAs.",
    "icon": "Sparkles",                     // Any Lucide icon name.
    "category": "headers",                  // Maps to widgetCategories or one of:
                                            //   headers / content / media / marketing / commerce
    "premium": false,                       // Tier-gated by proWidgets capability when true.
    "fields": [                             // The inspector's field list.
        { "key": "title",    "type": "string",   "label": "Headline", "required": true },
        { "key": "subtitle", "type": "textarea", "label": "Subhead" },
        { "key": "primaryCta",   "type": "string", "label": "Primary CTA label" },
        { "key": "primaryHref",  "type": "string", "label": "Primary CTA URL" }
    ],
    "componentImport": "@mero/theme-base",  // Optional. Defaults to the shared package
                                            //   for built-in widget types; falls back to
                                            //   `../components/sections/<Type>` otherwise.
    "componentName":   "Hero",              // Optional. Defaults to type.
    "aliases": ["hero", "banner"]           // Optional. Legacy section ids that should
                                            //   route to this widget at render time.
}
```

### Supported `fields[].type` values

| Type | Editor control | Stored as |
| --- | --- | --- |
| `text` / `string` | single-line input | string |
| `textarea` | multi-line input | string |
| `richtext` | multi-line input (markdown subset) | string |
| `number` | numeric input | number |
| `image` / `media` | media picker modal | URL string |
| `select` | dropdown (uses `options[]`) | string |
| `color` | color picker + hex input | hex string |
| `slider` / `range` | range slider with `min`/`max`/`step`/`unit` | number |
| `font` | curated font-family dropdown + custom input | CSS font-family stack |
| `button` | structured `{text, url}` editor | object |
| `buttons` | repeating `[{text, url}]` editor | object[] |
| `stats` | repeating `[{value, label}]` editor | object[] |
| `repeater` | typed array editor (uses `itemSchema[]`) | object[] |
| `json` | free-form JSON / array editor | any |
| `managed` | read-only "managed elsewhere" info card | (display-only) |

`itemSchema[]` (for `repeater`) can use a subset: `text` / `string` / `textarea` / `number` / `select` / `color` / `image` / `media`.

## 4. `pageSchema` — entry shape

```jsonc
{
    "slug": "home",                  // 'home' for /, otherwise routed at /<slug>
    "label": "Home",
    "sections": [
        { "id": "hero",     "type": "Hero" },
        { "id": "logos",    "type": "LogoStrip" },
        { "id": "features", "type": "FeatureBlocks" }
    ]
}
```

Every `type` referenced here must exist in `widgetCatalog` or `moduleSchemas`. The validator catches dangling refs.

## 5. `seedData` — initial content

```jsonc
{
    "pages": [
        {
            "slug": "home",
            "title": "Welcome",
            "sections": [
                {
                    "id": "hero",
                    "type": "Hero",
                    "data": {
                        "title": "Build something people brag about.",
                        "subtitle": "Subheadline copy goes here."
                    }
                }
            ]
        }
    ]
}
```

Every `slug` here must match a `pageSchema` slug. The activation flow runs once when the theme is first installed; subsequent edits go through the admin and don't touch seed data.

## 6. The editor bridge protocol

The visual editor talks to the public theme via `postMessage`. The contract is implemented in `@mero/theme-editor-bridge` and is versioned with the package.

### What you, the theme author, do

Mount the bridge once:

```tsx
// src/app/layout.tsx
import { EditorBridge } from '@mero/theme-base';
// (re-exports from @mero/theme-editor-bridge so you can import from either)

<body>
    {children}
    <EditorBridge />
</body>
```

Mark every editable surface:

```tsx
<section data-section-id="hero" data-section-type="Hero">
    <h1 data-editable="title">{data.title}</h1>
    <p  data-editable="subtitle">{data.subtitle}</p>
    <Button>
        <span data-editable="primaryCta">{data.primaryCta}</span>
    </Button>
</section>
```

That's it. The bridge handles hover outlines, click-to-select, live in-iframe text edits, navigation sync, and reload-on-save.

### Wire-protocol messages (FYI, you don't write these)

The bridge documents these in `@mero/theme-editor-bridge/protocol`:

- Outbound (theme → admin):  `mero-editor:ready` · `mero-editor:section-click` · `mero-editor:field-click`
- Inbound  (admin → theme):  `mero-editor:field-update` · `mero-editor:navigate` · `mero-editor:highlight` · `mero-editor:clear-highlights` · `mero-editor:reload`

When you upgrade `@mero/theme-editor-bridge`, follow semver — major bumps mean the admin and the theme must upgrade together.

## 7. Capabilities

A capability is a feature flag carried on the customer's package. The theme reads them via `useCapabilities()` and conditionally renders.

```tsx
import { useCapabilities, CapabilityGate } from '@mero/theme-base';

export function NewsletterSignup() {
    return (
        <CapabilityGate capability="forms" fallback={null}>
            <ContactForm formSlug="newsletter" />
        </CapabilityGate>
    );
}
```

Known capabilities (extend in `requiredCapabilities` / `optionalCapabilities`):

`pluginMarketplace · themeCodeEdit · visualThemeEditor · proWidgets · dashboardBranding · webhooks · collections · forms · analytics · auditLog · siteEditor · seoFull · aiContent · apiAccess · whiteLabel`

## 8. Plugin widgets

When a customer installs a plugin that ships widgets, those widgets show up in the editor's palette automatically (Phase 6.1). To render them on the public site (Phase 6.2), the plugin author ships their component as part of an npm package and the theme calls:

```tsx
import { registerPluginWidget } from '@mero/theme-base';
import CarouselPro from '@mero-plugin/carousel-pro';

registerPluginWidget('carousel-pro', 'carousel-pro:Carousel', CarouselPro);
```

If a plugin widget is on the page but no component is registered, the theme renders a placeholder visible only in edit mode — the public site shows nothing instead of breaking.

## 9. CLIs

| Command | What it does |
| --- | --- |
| `npm run theme:validate -- <slug>` | Lints `theme.json` against this spec. Required to be 0 errors before activation. |
| `npm run theme:validate -- --all` | Lints every theme in the workspace. |
| `npm run theme:build-registry -- <slug>` | Generates `widget-registry.gen.tsx` from `widgetCatalog`. Re-run after every catalog edit. |
| `npm run package-theme -- <slug>` | Bundles the theme into a `.mero-theme` archive ready for the admin's theme uploader. |

## 10. Versioning

| Version bump | What changed |
| --- | --- |
| **Patch** | Visual tweaks, bug fixes. No `theme.json` shape changes. Drop-in safe. |
| **Minor** | New optional fields in `theme.json`. New section components. Drop-in safe; existing sites keep rendering. |
| **Major** | Removed/renamed fields. New required keys. Changed pageSchema in incompatible ways. Customers must opt in. |

Be conservative with majors. Customers don't upgrade themes often, so a major bump means a content migration most of the time.

## 11. Don'ts

- Don't read directly from the database. Always go through the public API client (`getSiteData`, `getPage`, `getMenu`, `submitForm`).
- Don't write your own iframe protocol. Use `@mero/theme-editor-bridge`.
- Don't hardcode the API base URL. The client picks it up from runtime config.
- Don't ship widget icons that aren't in Lucide. The icon resolver falls back to `Square`, which looks broken in the palette.
- Don't bypass capability gates. The activation flow assumes themes honour their declared `requiredCapabilities`.
- Don't ship secrets in `theme.json`. The file is served to the admin verbatim.
