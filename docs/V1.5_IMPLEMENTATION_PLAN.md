## 1. Background — why this plan exists

An audit against `backend/src/config/packages.ts` found that every boolean capability (`siteEditor`, `analytics`, `webhooks`, `pluginMarketplace`, `visualThemeEditor`, `dashboardBranding`, etc.) is fully implemented and gated. Three items are sold on the pricing page but only partially built:

| Feature | Sold on | Backend | Frontend | Verdict |
|---|---|---|---|---|
| White-label (remove Mero branding) | Personal Custom, Org Enterprise, Org Custom | Gate exists | No UI, no implementation | Not shipped |
| API key management | Personal Professional+, Org Enterprise+ | Gate exists | No UI, no key model, no middleware | Not shipped |
| AI Content Studio | Personal Professional+, Org Enterprise+ | `/ai/generate` endpoint + blog assistant | No standalone page | Partial |

In addition, a few enforcement paths (team seat limit on invite, storage limit on media upload) need a smoke test to confirm the guards fire. Support tiers (email / priority / dedicated) are metadata-only today — that's intentional, not a bug, but the plan notes an optional in-app surface for the top tier.

---

## 2. Priority and rough effort

The order below is deliberate: white-label is the smallest win and raises the perceived "premium" feel of Enterprise; API keys unlock a real developer audience; AI Studio is the biggest user-visible lift.

1. **White-label branding** — ~0.5 day
2. **API key management** — ~2 days
3. **AI Content Studio page** — ~2 days
4. **Enforcement smoke tests** — ~0.5 day
5. **Support-tier surface** (optional) — ~0.5 day

Total: one focused sprint.

## 3. White-label branding

### Goal

Users on tiers with `hasWhiteLabel: true` (Personal Custom, Org Enterprise, Org Custom) can hide every "Powered by Mero CMS" touchpoint in the admin and set their own footer copy. Lower tiers keep the badge, which reinforces the upgrade path without being aggressive.

### What to build

A single settings surface with two keys and one gate-driven component replacement.

**Backend**

- No new capability needed — `hasWhiteLabel` is already on the `CmsPackage` shape.
- Extend `getCapabilities()` return (or the `/packages/usage` response) to expose `hasWhiteLabel` on the frontend — it's already bundled via `limits.hasWhiteLabel`, so nothing new here.
- Two new setting keys in the existing KV store:
  - `whitelabel_hide_powered_by` (boolean, default `false`)
  - `whitelabel_footer_text` (string, e.g. "© 2026 Your Company")
- Gate writes to these keys on `PATCH /settings` behind `PackageLimit.WHITE_LABEL`, matching the `admin-branding` pattern.

**Frontend**

- New component `<PoweredByBadge />` used everywhere today — in the admin footer, the customize page footer, the login screen. Component reads `hasWhiteLabel` + `whitelabel_hide_powered_by` from `CapabilitiesContext` and `SettingsContext`; renders nothing if both are set, renders the badge otherwise.
- New settings tab **Branding → White Label**, visible only when `hasWhiteLabel === true`. Toggle for "Hide Powered by Mero CMS" and a text field for custom footer copy.
- Upgrade nag: on lower tiers the tab still appears but shows an Enterprise upsell card instead of the form.

### Files to touch

```
frontend/src/components/ui/PoweredByBadge.tsx   # NEW

frontend/src/app/(admin)/dashboard/settings/page.tsx  # register tab
frontend/src/app/(admin)/layout.tsx             # replace footer branding with <PoweredByBadge />

```

### Acceptance criteria

- On `personal-basic`: badge visible everywhere, tab shows upsell.
- On `org-enterprise`: tab shows form; toggling hides the badge immediately (no reload).
- Switching from Enterprise back to Premium: badge re-appears; the stored `whitelabel_hide_powered_by` becomes ignored but isn't deleted.

## 4. API key management

### Goal

Let developers on Pro/Enterprise/Custom generate and revoke API keys and hit the existing JWT-protected REST endpoints with a header auth token instead of a browser session. This is what "Direct API Access" on the pricing page promises.

### Data model

Add one table:

```prisma
// prisma/modules/api-keys.prisma
model ApiKey {
  id         String   @id @default(cuid())
  name       String                   // user-given label, "Production ingest key"
  tokenHash  String   @unique         // sha256 of the real token (never stored raw)
  prefix     String                   // first 8 chars, shown in list for identification
  userId     String                   // owner
  user       User     @relation(fields: [userId], references: [id], onDelete: Cascade)
  scopes     String[] @default([])    // e.g. ["pages:read", "posts:write"] — enforced at the guard
  lastUsedAt DateTime?
  expiresAt  DateTime?                // null = never
  revokedAt  DateTime?                // soft revoke
  createdAt  DateTime @default(now())

  @@index([userId])
}
```

Raw token shape: `mk_live_<32-char-base62>`. Hashed on write, shown **once** on creation.

### Backend

New module `backend/src/api-keys/`:
- `api-keys.service.ts` — `create`, `list`, `revoke`, `validate(token)` that hashes + looks up + bumps `lastUsedAt`.
- `api-keys.controller.ts` — `POST /api-keys`, `GET /api-keys`, `DELETE /api-keys/:id`. All three require `PackageLimit.API_ACCESS`.
- `api-key-auth.guard.ts` — accepts `Authorization: Bearer mk_live_…`, resolves to a user, and reuses the existing `PermissionsGuard` for scope checking. Applied alongside `JwtAuthGuard` via a composite guard so either session cookie or API key works.

### Frontend

- `/dashboard/api-keys` page (gated by `hasApiAccess`):
  - List of existing keys with prefix, name, last-used, revoke button.
  - "New API Key" modal: name input, scope multi-select, optional expiry date. On submit, shows the full token **once** with a big "Copy" button and warning that this is the only time it'll be visible.
  - Revoke confirmation modal.
- Sidebar nav entry **Developers → API Keys**, gated by `hasApiAccess`.
- Optional: a `/dashboard/api-docs` stub that lists the public REST surface (can be a Markdown doc rendered with the existing `/dashboard/docs` shell).

### Files to touch / create

```
prisma/modules/api-keys.prisma           # NEW
backend/src/api-keys/                    # NEW module
backend/src/api-keys/api-keys.controller.ts
backend/src/api-keys/api-keys.service.ts

backend/src/api-keys/api-key-auth.guard.ts
backend/src/app.module.ts                # register module
backend/src/pages/pages.controller.ts    # update guards to accept API-key auth
                                         # (same pattern for posts, collections, forms)
frontend/src/app/(admin)/dashboard/api-keys/page.tsx  # NEW
frontend/src/components/layout/Sidebar.tsx            # add nav entry
```

### Acceptance criteria

- Creating a key shows the token once, stores the hash, and subsequent `GET /api-keys` returns only prefix + metadata.
- `curl -H "Authorization: Bearer mk_live_…" /pages` returns the same data as the admin session does.
- Revoking invalidates the key immediately (check `revokedAt IS NOT NULL` in `validate()`).
- On `personal-premium`: the page shows a "Professional plan required" upsell card instead of the form.

## 5. AI Content Studio page

### Goal

Turn the existing backend `/ai/generate` endpoint into a standalone Content Studio that generates more than just blog draft ideas — page copy, meta descriptions, image alt text, email subject lines, FAQ answers, product descriptions. The blog's `AiAssistant` component stays as-is; this is an addition.

### Backend

- Extend `AiService.generate(prompt, context)` to accept a `preset` field: `blog_draft | page_copy | seo_meta | alt_text | email_subject | faq | product_description`. Each preset seeds a curated system prompt and output format.
- Add `POST /ai/templates` returning the preset list with labels and example prompts, so the frontend can render a grid without hardcoding.
- Rate limit per user via a simple in-memory token bucket (e.g., 60 generations/hour on Professional, 200/hour on Enterprise). Stored as `ai_usage_<userId>` in the settings KV with a rolling window, or a dedicated `AiUsage` Prisma model if you want persistence.

### Frontend

- `/dashboard/ai-studio` page:
  - Left column: preset picker (cards with icons), e.g. "Write a blog intro", "Generate SEO meta", "Summarize this page for alt text".
  - Main area: prompt input + context input (optional — for "polish this text" flows) + model choices if the backend supports multiple.
  - Right column: history of last 20 generations for this user (reuse the KV-stored audit trail or a light `AiUsage` table).
  - "Copy", "Insert into clipboard", and "Save as snippet" actions on each result.
- Sidebar entry **Tools → AI Studio**, gated by `aiEnabled`.
- Quick-use buttons in existing editors: "Generate meta description" in the SEO tab of the page editor, "Suggest alt text" in MediaLibrary. Each one calls the AI preset with the correct context instead of opening a generic modal.

### Files to touch / create

```

prisma/modules/ai.prisma                 # NEW (optional — for persistent history/limits)
frontend/src/app/(admin)/dashboard/ai-studio/page.tsx   # NEW

```

### Acceptance criteria

- All 7 presets return sensible completions.
- Rate limiting surfaces a clear "You've hit your hourly AI limit — resets at HH:MM" toast, not a 500.
- On `personal-premium`: the nav entry is hidden; direct-navigating to `/dashboard/ai-studio` shows the upsell state.
- The blog's existing `AiAssistant` keeps working without changes (no regression).

## 6. Enforcement smoke tests

### Goal

Confirm the guards that were written actually fire in production code paths — not just in unit logic.

### Checks to run manually

- **Team seat limit**: On a `personal-basic` plan (`teamLimit: 1`), go to `/dashboard/team`, try to invite a second member. Expected: `ForbiddenException` rendered as a toast ("Team seat limit reached"). If it doesn't fire, check that `POST /users/invite` has `@RequireLimit(PackageLimit.TEAM_SIZE)`.
- **Storage limit**: On a `personal-basic` plan (`storageLimitGB: 5`), upload media totalling >5GB. Expected: 5GB'th upload is rejected with a storage-limit error. If not, check `/media/upload` is behind the guard and that `getUsage().usage.storageGB` is actually aggregating file sizes.
- **Theme count**: On a `personal-basic` plan, try to `POST /themes/upload`. Expected: rejected — this should already be enforced by `THEME_COUNT`.
- **White-label writes**: after shipping section 3, confirm a `personal-basic` user cannot `PATCH /settings` with `whitelabel_hide_powered_by: true`.

### Files possibly to touch

```
backend/src/users/users.controller.ts     # verify TEAM_SIZE on invite
backend/src/media/media.controller.ts     # verify STORAGE on upload
```

If any check fails, the fix is usually a missing `@RequireLimit(...)` and `@UseGuards(PackageEnforcementGuard)` on the write route.

### Acceptance criteria

All four checks above pass or get patched within the same PR.

## 7. Support-tier surface (optional, low priority)

### Goal

Give top-tier customers a visible in-app acknowledgement that they're on Dedicated support — ticketing is still a human process, but a contact channel widget reinforces the tier.

### What to build

- A floating **Help** button (the existing `<HelpFAB />` is already there). When clicked on:
  - `email` tier: opens a form that emails `support@` (already works).
  - `priority` tier: form plus an "Expected response within 24h" banner.
  - `dedicated` tier: also shows the customer's dedicated account manager (`settings.support_manager_name`, `settings.support_manager_email`).
- No backend changes beyond adding three optional settings keys: `support_manager_name`, `support_manager_email`, `support_slack_channel`.

### Files

```
frontend/src/components/ui/HelpFAB.tsx     # extend with tier-aware content
backend/src/settings/settings.controller.ts  # no change — new keys ride on the generic PATCH
```

### Acceptance criteria

Dedicated-tier users see their account manager's name and email inside the HelpFAB. Email-tier users see the normal form.

## 8. Out of scope (explicit)

- **Custom tier custom services** (on-premise, custom integrations, training, 24/7 support): these are sold, not built. No engineering work here; document them in `PRODUCT.md` as fulfilment items.
- **Ticketing system**: punts to Zendesk/Linear/Intercom if needed later. Not a v1.5 concern.
- **Public API rate limiting per plan**: v1.6 — once API keys exist, add per-plan limits (e.g. Pro = 10k/day, Enterprise = 100k/day).
- **API billing metering**: v2.0 — would need metered billing infra.
